A — Gestion des Missions & Planning
TICKET #1 Draft & Publication des Missions 1 500 €
Description fonctionnelle

Ajouter un système de brouillon (draft) lors de la création et modification des missions. L'admin peut effectuer plusieurs modifications successives sans déclencher de notifications. Un bouton « Publier les changements » envoie les notifications (push, e-mail) uniquement aux agents impactés par les modifications. Si la modification affecte toute la mission, tous les agents assignés sont notifiés.

Périmètre technique
Admin — Planning.tsx Backend — events.ts Backend — Prisma schema Backend — socket.ts
Impacts techniques

• Ajout d'un champ isDraft: Boolean + table EventDraftChanges dans le schéma Prisma pour stocker les changements en attente.
• Modification de la route PUT /api/events/:id pour supporter le mode draft (sauvegarde sans émission socket).
• Nouvelle route POST /api/events/:id/publish : calcul du diff entre état publié et brouillon, identification des agents affectés, envoi ciblé des notifications push/email.
• Intégration d'un service d'envoi e-mail (Nodemailer / SendGrid) pour les notifications par mail.
• Mise à jour du composant Planning.tsx : indicateur visuel « Brouillon non publié », bouton Publier avec confirmation.

Justification du prix

Complexité élevée : nécessite un système de diff pour comparer les états, un mécanisme de notifications ciblées (push + email), l'intégration d'un service e-mail, et une refonte partielle du flux de sauvegarde des événements. Estimation : ~4 jours de développement.

TICKET #7 Gestion des Heures de Pause dans les Missions 350 €
Description fonctionnelle

Lors de la création d'un événement, l'admin peut saisir un nombre d'heures de pause. Cette durée est automatiquement déduite du total prévu. Exemple : un créneau 8h–14h avec 1h de pause = 5h de travail effectif au lieu de 6h. Cette donnée impacte le calcul des heures prévues dans les récaps et les comparaisons pointé vs prévu.

Périmètre technique
Admin — Planning.tsx Agent — MyPlanning.tsx Backend — schema.prisma, events.ts
Impacts techniques

• Ajout du champ breakHours: Float? sur le modèle EventShift ou Event dans Prisma.
• Mise à jour du formulaire de création/édition d'événement avec un sélecteur d'heures de pause.
• Modification du calcul totalPrévu = (endTime - startTime) - breakHours dans HoursTracking et Recap.
• Affichage de la pause dans le détail créneau (côté agent et admin).

Justification du prix

Complexité faible-moyenne : ajout d'un champ simple mais avec impact sur plusieurs calculs existants (heures prévues, comparaisons, récaps). Estimation : ~1 jour.

Justification du prix

Complexité moyenne : ajout d'un champ et d'une logique de validation croisée entre les entités User, Event et Site. Impact sur le calcul financier du Recap. Estimation : ~1.5 jours.
