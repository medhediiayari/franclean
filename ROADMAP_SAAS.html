2. Stratégie Multi-Tenancy

Isolation par instance vs. base partagée — choix et conséquences

✅ Option A — Instance par Client (Recommandée)

  • 1 conteneur Docker + 1 BDD PostgreSQL par client
  • Isolation totale des données (conformité RGPD native)
  • Chaque client a son sous-domaine : client1.votreplateforme.com
  • Mise à jour indépendante possible
  • Scaling horizontal simple
  • Facile avec l'architecture Docker actuelle

Coût serveur : ~5-15€/mois par instance (VPS optimisé)

⊘ Option B — BDD Partagée (Non retenue)

  • 1 seule BDD avec tenantId sur chaque table
  • Modification lourde du schema Prisma
  • Risque de fuite de données entre tenants
  • Toutes les requêtes doivent filtrer par tenant
  • Un bug = tous les clients impactés
  • Scaling vertical uniquement

Moins cher à petite échelle, mais bien plus risqué

⚠ Point Critique Détecté : Photos Base64 en BDD

  • Actuellement les photos de pointage (check-in/out) sont stockées comme chaînes base64 directement dans PostgreSQL (colonne checkInPhotoUrl)
  • Chaque photo JPEG 0.8 qualité ≈ 200-500 Ko en base64
  • Avec 30 agents × 2 photos/jour = 12-30 Mo/jour dans la BDD
  • En 1 mois : ~360-900 Mo de photos dans PostgreSQL
  • En 1 an : ~4-10 Go — la BDD devient lente

Impact : Requêtes GET /api/attendance retournent TOUTES les photos en JSON → transfert réseau énorme, mémoire Node saturée si beaucoup de records.

🔧 Solution Recommandée (Pré-requis SaaS)

  • Migrer les photos vers stockage objet S3 (Hetzner Object Storage / MinIO / AWS S3)
  • En BDD, stocker uniquement l'URL : https://s3.../tenant1/photo-xxx.jpg
  • Upload direct depuis le frontend vers S3 via presigned URL
  • Compression côté client avant upload (déjà fait : JPEG 0.8)
  • CDN Cloudflare devant S3 pour cache des images
  • Gain : BDD reste < 100 Mo même après des années
  • Gain : Transfert API réduit de 90% (URLs au lieu de blobs)

Impact : RAM Node.js réduite de 40%, temps de réponse API divisé par 5 sur les listes de pointages.

📡 Métriques de Déclenchement Auto-Scale

  • RAM > 80% sur un nœud → alerte + provisionnement nouveau nœud
  • CPU > 70% soutenu 5 min → scale up vertical ou horizontal
  • Temps de réponse API > 500ms (p95) → investigation + scaling
  • Connexions Socket.IO > 80% du max → redistribution
  • Stockage BDD > 5 Go par instance → alerte (probablement photos non migrées)
  • Queue d'emails > 100 → scale du service SMTP

Toutes ces métriques sont collectées par Prometheus et visualisées dans Grafana avec alertes Slack/email.

# Monitoring daemon (cron chaque 5 min)

🌐 Routage Réseau (Traefik)

  • Traefik comme reverse proxy / load balancer
  • Détection automatique des containers Docker (labels)
  • Routage par sous-domaine : client1.plateforme.com → container frontend-client1
  • API proxy : /api/* → container api-client1
  • WebSocket proxy : /socket.io/* → container api-client1 (upgrade header)
  • SSL automatique via Let's Encrypt (wildcard cert via Cloudflare DNS challenge)
  • Réseau Docker isolé par tenant : tenant-{ID}-net
  • Seul Traefik est exposé publiquement (ports 80/443)

🔒 Sécurité Réseau

  • Chaque instance dans son propre réseau Docker isolé
  • PostgreSQL jamais exposé sur l'Internet (réseau interne uniquement)
  • Firewall UFW : seuls ports 80, 443, 22 ouverts
  • Cloudflare WAF : protection XSS, SQL injection, DDoS
  • Rate limiting : 100 req/min par IP par instance (Traefik middleware)
  • JWT secret unique par instance (généré au provisionnement)
  • Pas de communication possible entre tenants (isolation réseau L3)
  • SSH hardened : clé uniquement, port non-standard, fail2ban

🛡 Backups & Disaster Recovery

  • Quotidien (2h AM) : pg_dump de chaque instance → gzip → chiffrement AES-256 → upload vers Storage Box
  • Rétention : 7 quotidiens + 4 hebdomadaires + 3 mensuels
  • Test de restauration : automatisé le 1er de chaque mois (restore dans un container temporaire + vérification intégrité)
  • RTO (Recovery Time) : < 30 minutes
  • RPO (Recovery Point) : < 24 heures (quotidien)
  • Option WAL streaming : pour RPO < 5 min (plan Entreprise)
  • Photos S3 : réplication automatique cross-datacenter (Hetzner)
  • Taille backup : ~5-20 Mo/instance (sans photos en BDD)

📊 Stack de Monitoring

  • Prometheus : collecte métriques toutes les 15s (CPU, RAM, réseau, requêtes API, connexions DB, sockets)
  • Grafana : dashboards par instance + vue globale
    • Dashboard "Overview" : toutes instances en un coup d'œil
    • Dashboard "Instance" : détail par tenant
    • Dashboard "Business" : inscriptions, churn, MRR
  • Loki : logs centralisés de tous les containers
  • Uptime Kuma : ping health check chaque 60s sur /api/health de chaque instance
  • Alertes : Slack + Email si :
    • Instance down > 2 min
    • RAM > 80% pendant 5 min
    • Error rate > 1% pendant 3 min
    • Backup échoué

10. Aspects Légaux, RGPD & Conformité

Obligations légales pour un SaaS B2B en France/Europe

📜 Documents Juridiques Requis

  • CGU — Conditions Générales d'Utilisation
  • CGV — Conditions Générales de Vente
  • Politique de confidentialité (RGPD)
  • DPA — Data Processing Agreement
  • Mentions légales (obligatoire FR)
  • SLA — Service Level Agreement (plan Entreprise)
  • Politique de cookies

🔐 Conformité RGPD

  • Isolation des données : ✅ native (1 BDD/client)
  • Droit de suppression : script de purge par instance
  • Droit d'export : export JSON/CSV des données
  • Consentement : opt-in explicite à l'inscription
  • Registre des traitements : documentation interne
  • DPO : désigner un responsable si >250 employés
  • Hébergement EU : Hetzner (Allemagne) ✅

⚖ Budget Juridique Recommandé

Rédaction CGU/CGV/RGPD par un avocat spécialisé : 1 500 — 3 000€ (one-time). C'est un investissement indispensable avant le lancement public. Des templates existent pour réduire le coût initial.
